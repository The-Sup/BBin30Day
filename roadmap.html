Web Fundamentals — How the Web Works for Hackers
HTTP, cookies, sessions, headers, request/response lifecycle
MindsetCore
▼
What & Why

You cannot exploit what you don't understand. Every web vulnerability is an HTTP request gone wrong. Understanding sessions, cookies, headers, and the request lifecycle is the foundation every advanced technique is built on.

Topics to Master Today
  • HTTP methods: GET, POST, PUT, DELETE, PATCH, OPTIONS — and when each is used (APIs especially)
  • Status codes that matter: 200, 201, 301/302, 400, 401, 403, 404, 429, 500 — and what each signals to a hunter
  • Headers to know: Authorization, Cookie, Content-Type, Origin, Referer, X-Forwarded-For, Host, Content-Security-Policy, Access-Control-Allow-Origin
  • Session management: How cookies work, HttpOnly/Secure flags, SameSite attribute — each flag affects a different attack class
  • CORS: What it is, why misconfigurations matter, what an open CORS header actually lets an attacker do
  • Burp practice: Intercept a login, see the POST body. Change a cookie value. Observe a 302 redirect.
Burp Exercise
# In Burp Repeater — modify these on any site and observe responses: 1. Change GET → POST on a search endpoint 2. Add: X-Forwarded-For: 127.0.0.1 (bypass IP-based restrictions) 3. Add: Origin: https://evil.com (test CORS) 4. Change Content-Type: application/json → text/xml 5. Modify a cookie value — observe if validation breaks

Choose Your Program — How to Pick a Target That Pays
Program selection strategy, scope analysis, reading policy documents
Strategy
▼
What & Why

Program selection is where most beginners make their first fatal mistake — they either pick a huge target with millions of hunters (brutal competition) or a tiny scope with nothing to find. Picking the right program at the right time is a skill in itself.

Selection Criteria
  • Wide scope beats high payout early on — you need surface area to find things
  • Response time: Sort HackerOne programs by "Average Bounty" AND "Average Response Time" — slow responders kill momentum
  • Recently launched programs: Less competition, lower duplicate rate
  • Asset types you're comfortable with: web app vs API vs mobile — pick one and go deep
  • Read the policy document completely: Note every out-of-scope item and what specific vulnerabilities they won't pay for (rate limiting, self-XSS, etc.)
  • Your starter picks: HackerOne — US DoD (very wide scope, low competition), any mid-size SaaS with recent launch

The Hunter Mindset — How Top Earners Think
Impact-first thinking, where bugs actually live, patience vs frustration
Mindset
▼
What & Why

The biggest skill gap isn't technical — it's mental. Top hunters think in business impact, not just technical payloads. They ask "what could go wrong here for the company?" before "what payload breaks this?" This shift changes what you test and what you find.

Core Mental Frameworks
  • Think like the developer, not the attacker: What shortcuts did they take? What did they forget to validate? What's the "happy path" vs edge cases?
  • Feature → Attack surface mapping: Every new feature = new attack surface. File upload? Think path traversal, stored XSS, SSRF. Export? Think IDOR, data exposure. SSO? Think account takeover.
  • Where bugs concentrate: New features (less tested), mobile APIs (often less secure than web), admin functionality, payment flows, file operations, search, import/export
  • The 80/20 of bug bounty: 80% of paid bugs are IDOR, SQLi, XSS, SSRF, auth bypass. Master these first.
  • Duplicate avoidance: Check program's hacktivity before testing obvious endpoints. Test less-obvious flows instead.
  • Your daily habit: 30 min reading disclosed reports → 2hr active testing → 30 min documenting

ReconSecrets
▼
What & Why

JavaScript files are goldmines. Developers embed API keys, internal endpoints, auth tokens, and business logic directly in client-side JS. This is one of the highest-signal, lowest-competition recon steps because most hunters skip it.

JS Recon Pipeline
# Extract all JS files from target cat crawl.txt | grep "\.js$" | sort -u > jsfiles.txt cat wayback.txt | grep "\.js$" >> jsfiles.txt # Download all JS files cat jsfiles.txt | xargs -I{} wget -q {} -P ./js_files/ # Scan for secrets with secretfinder python3 SecretFinder.py -i https://target.com -e -o results.html # Grep for common secrets manually grep -rEi "api[_-]?key|secret|token|password|auth|bearer|aws|s3" ./js_files/ # Find hidden endpoints in JS cat jsfiles.txt | xargs -I{} curl -s {} | grep -oE "(['\"])/[a-zA-Z0-9_/.-]+" | sort -u # Use LinkFinder for structured extraction python3 linkfinder.py -i https://target.com -d -o results.html
What to Flag
  • Hardcoded API keys (AWS, Google Maps, Stripe, Twilio, SendGrid)
  • Internal API endpoints not exposed in the main site map
  • Debug/test endpoints left in production JS
  • Authentication logic — look for JWT handling, session checks

IDOR — The #1 Bug Bounty Finding
Object reference manipulation, horizontal + vertical escalation
VulnHigh Payout
▼
What & Why

IDOR (Insecure Direct Object Reference) is the most consistently paid vulnerability in bug bounty. Why? It's everywhere, developers constantly miss it, and the impact is direct — you access other users' data. It's the "find and report" vulnerability.

IDOR Attack Vectors
  • Numeric IDs: /api/user/1001 → try 1000, 1002, 999 — can you see other users?
  • GUIDs: Not safe by default. Find a GUID in response, try in another account's context
  • Encoded references: Base64 decode parameters — often hides user_id=123
  • API endpoints: POST /api/order/delete {"order_id": "12345"} — change ID to another user's order
  • File downloads: /download?file=report_user123.pdf → enumerate other users' reports
  • Autorize workflow: Log in as User A → copy session cookie to Autorize → browse as User B → Autorize auto-tests every request with User A's token
Testing Methodology
# 1. Create two accounts (attacker + victim) # 2. As victim: perform action (create order, upload file, update profile) # 3. Note the object ID in the request # 4. As attacker: replay the request with victim's object ID # Example: Profile access IDOR GET /api/v1/users/victim_id/profile Authorization: Bearer attacker_token # If response contains victim's data = IDOR confirmed # Document: request, response, both account contexts
Lab
Complete all IDOR labs on PortSwigger Web Security Academy (free). Then test your chosen program's API endpoints for IDOR using two test accounts.

VulnCritical
▼
ATO Attack Surface
  • Password reset token predictability: Request multiple resets — is the token random? Sequential? Time-based?
  • Host header injection in reset email: Change Host header → reset link goes to your server with victim's token
  • JWT attacks: Change alg to "none" → remove signature. Or crack weak HS256 secret with hashcat
  • OAuth state parameter missing: CSRF on OAuth flow = account takeover
  • 2FA bypass: Does the app enforce 2FA on API endpoints? On mobile endpoints? Test all auth paths independently
  • Username enumeration: Different error messages for valid vs invalid usernames → enables targeted attacks

Business Logic Flaws — The Unautomatable Bugs
Price manipulation, workflow bypass, race conditions
VulnManual
▼
What & Why

Business logic bugs can't be found by scanners — they require understanding what the application is supposed to do and finding cases where it doesn't enforce those rules. These bugs are often Critical and almost never duplicated because they require deep application knowledge.

Test These Scenarios
  • Price manipulation: Negative quantities, changing price parameters in POST body, coupon stacking, currency confusion
  • Workflow bypass: Can you skip step 2 of a 3-step checkout? Access step 3 directly?
  • Race conditions: Send 50 simultaneous requests to redeem a single-use coupon — does it apply multiple times?
  • Privilege escalation via parameter: Is role/admin/is_admin sent client-side? Try changing it.
  • Mass assignment: In registration POST body, add fields like "role":"admin" — does the server accept it?
  • Discount abuse: Apply 100% discount code. Apply expired promo. Apply codes beyond their limit.
Race Condition with Turbo Intruder
# In Burp: right-click request → Extensions → Turbo Intruder # Use the race condition template # Send 50 parallel requests simultaneously # Watch for: different success responses, multiple DB writes

Writing Bug Reports That Get Paid — Not Closed
Report structure, impact narrative, CVSS, reproduction steps
Reporting
▼
What & Why

A valid bug with a poor report gets downgraded or closed. The triage team receives hundreds of submissions — yours must be impossible to misunderstand. Clear reproduction steps, undeniable impact, and professional communication are the difference between $0 and $2,000.

The Anatomy of a Paid Report
Title: [IDOR] Any authenticated user can access and modify other users' private billing information via /api/v2/billing/{user_id} Summary: An IDOR vulnerability in the billing API allows authenticated users to read and modify billing details of any other user by manipulating the user_id parameter. No additional authorization is performed server-side. Severity: High (CVSS 8.1) Steps to Reproduce: 1. Log in as User A (attacker@test.com) 2. Navigate to Billing Settings — intercept with Burp Suite 3. Observe GET request: /api/v2/billing/12345 (your user_id) 4. Send to Repeater. Change 12345 to 12344 (victim's user_id) 5. Evidence: [Screenshot 1: Request with victim user_id] [Screenshot 2: Response showing victim's billing data] [Screenshot 3: Victim's actual account showing matching data] Remediation: Enforce server-side authorization on every /api/v2/billing/{id} request. ✓
Vulnerability Chaining — Turn Low Bugs Into Criticals
Combining multiple weaknesses for maximum impact
AdvancedHigh Payout
▼
What & Why

A Self-XSS alone is usually P4 ($0). A CSRF that triggers the Self-XSS is P2 ($500-2000). Add an Open Redirect to deliver the CSRF payload and you have a P1 account takeover chain. Chaining is how you turn three $0 findings into one $5,000 report.

Classic Chains to Know
  • Self-XSS + CSRF = Stored XSS: Use CSRF to make victim trigger the XSS in their own profile
  • Open Redirect + OAuth = Account Takeover: Redirect OAuth token to attacker domain
  • SSRF + Internal Redis = RCE: SSRF to internal Redis → write cron job → command execution
  • Path Traversal + File Write = RCE: Write webshell via path traversal
  • Subdomain Takeover + Stored XSS = Session Hijack: Host malicious content on taken-over subdomain
  • IDOR + PII + Unauthenticated Access = Critical Data Breach: IDOR on endpoint that's also missing auth
Practice Task
Take your list of lower-severity potential findings. For each one, ask: what other weakness could I chain this with to escalate impact? Map possible chains in your notes before testing them.

API Security Testing — Where Modern Bugs Live
REST, GraphQL, hidden endpoints, mass assignment
VulnTools
▼
What & Why

Modern apps are 80% API. The web UI is just a pretty front-end — the real logic is in the API. API endpoints often have weaker authentication, less input validation, and better attack surface density than the web front-end.

API Testing Checklist
  • Find the API docs: /api/docs, /swagger.json, /api-docs, /openapi.json — read every endpoint
  • GraphQL introspection: Run query to dump full schema, find hidden mutations
  • HTTP method switching: GET /api/users → try DELETE, PUT, PATCH
  • Version downgrade: /api/v3/user → try /api/v1/user — older versions often lack auth
  • Mass assignment: Add unexpected fields to POST bodies — "role":"admin", "verified":true
  • Missing auth on actions: Does deleting via API require auth? Test without Authorization header.

# Test if introspection is enabled {"query": "{__schema{types{name}}}"} # Full schema dump {"query": "{__schema{queryType{name} mutationType{name} types{name kind fields{name args{name type{name kind ofType{name kind}}}}}}}"} # Use GraphQL Voyager to visualize the schema # Look for: admin mutations, delete operations, unauthenticated queries

Handling Triage — Communication That Preserves Reputation
Duplicates, N/A responses, severity disputes, professional pushback
ReportingMindset
▼
The Responses You'll Get (and How to Handle Them)
  • Duplicate: Happens constantly. Don't argue. Ask if you can see the original once resolved. Learn what was already found and move on faster.
  • N/A — out of scope: Re-read scope carefully. If you genuinely believe it's in scope, politely quote the relevant policy section. Don't argue without evidence.
  • Severity downgrade: Calculate CVSS yourself. If they rate P3 and you believe P2, write a factual rebuttal explaining real-world impact. Use concrete examples.
  • Needs more info: Respond within 24 hours. Provide a more detailed PoC, video recording, or alternative reproduction steps.
  • Won't fix / Informational: Accept gracefully. Note it for your portfolio. Some programs pay for informational findings on VDPs.
Never Do This
Never threaten disclosure. Never be rude to triagers. Never publicly disclose before the program's disclosure timeline. Never exploit a finding beyond what's necessary to prove impact. These behaviors permanently destroy your reputation in the community.
